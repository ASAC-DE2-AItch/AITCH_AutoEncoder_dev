## 작업 방식
- **노트북 전달 방식**: Claude가 완성된 `.ipynb`를 제작하고, 출력을 비운 깨끗한 노트북을 전달한다. **실행은 사용자가 로컬 venv에서** 한다. (*클라우드 미러에서 자체 검증(실행 재현) 미실행)
- **버전 관리**: 피처 엔지니어링, 모델, EDA 방식 등이 변경될 때(코드 버그 수정 제외) 기존 파일을 수정하지 않고 새 파일을 생성할 것 (예: `modeling_v2.ipynb`, `modeling_v3.ipynb`)
- **완료 후 자동 작업**: 사용자가 "컴파일 완료 됐어!"라고 보내면, (1) 해당 버전의 README와 REPORT 파일을 생성하고, (2) 다음에 무엇을 왜 어떻게 할 것인지 텍스트로 제안할 것
- **README / REPORT 역할 구분**:
  - **README**(`<버전>_README.md`)는 **살아있는 문서** — 노트북 사용법 + 누적 실험 로그 + 스냅샷 인덱스. 코드·결과가 바뀔 때마다 이 파일만 갱신한다(이름 고정).
  - **REPORT**는 **스냅샷(고정)** — 그 시점 분석 결과를 기록하고, 이후 마일스톤이 나와도 수정하지 않는다(분석이 바뀌면 새 REPORT).
- **REPORT 파일 명명·위치 규칙**: 스냅샷 REPORT는 **`<버전>/REPORT/` 하위 폴더**에 `<버전>_REPORT_<NN>_<마일스톤>.md`로 둔다 — 순번(01, 02…) + 마일스톤 태그. 마일스톤 배치마다 **새 번호 파일**을 만들고 제목도 그 배치로 고정한다(**제목에 마일스톤을 누적하지 않음**). 예: `modeling_v8/REPORT/modeling_v8_REPORT_01_M0.md`, `…_02_M1.md`. README·노트북·PLAN은 버전 폴더 루트에 두고, 누적 로그·버전 비교표·스냅샷 인덱스는 README에 모은다.
